# Packaging & Releases

How to build distributables, cut a release, and sign the binaries. For running from source, see [development.md](development.md).

## Packaging

Build unsigned distributable(s) locally:

```bash
npm run dist        # both platforms (only works on the current OS)
npm run dist:mac    # macOS: .dmg + .zip (x64 and arm64)
npm run dist:win    # Windows: NSIS installer + .zip (x64)
```

Output goes to `release/`. Unsigned builds run fine for local testing; macOS will show a Gatekeeper warning and Windows may show a SmartScreen prompt.

The Windows build emits **both** an NSIS installer (`.exe`) and a portable **`.zip`**. The zip is a no-installer fallback for locked-down machines: NSIS extracts and loads helper DLLs (`System.dll`) from `%TEMP%` at startup, and corporate endpoint security (AppLocker/WDAC DLL rules, Defender ASR, EDR) can block that, causing the installer to fail — sometimes silently. The zip has no installer and no temp extraction: unzip it anywhere the user can write and run `Central Tracking.exe` in place. Note that the zip does **not** set up the `ct` CLI wrapper or PATH entry (the installer does that); set those up manually if needed.

## Releases

Releases are built by GitHub Actions ([`.github/workflows/release.yml`](../.github/workflows/release.yml)). Push a version tag to trigger a build on both platforms, sign/notarize the artifacts, and publish a GitHub Release:

```bash
git tag v1.0.0
git push origin v1.0.0
```

The workflow produces signed artifacts only when the signing secrets are configured (see below). Without them it builds and uploads unsigned artifacts.

Every release also publishes:

- **`SHA256SUMS.txt`** — SHA-256 hashes of all distributables, for integrity checks.
- **Build-provenance attestations** — generated by [`actions/attest-build-provenance`](https://github.com/actions/attest-build-provenance), cryptographically tying each binary to the GitHub Actions run that built it (via Sigstore). These give downloaders a verifiable chain of custody even when the binaries aren't code-signed.

## Verifying a download

Even without code signing, users (and corporate IT) can confirm a binary is authentic and untampered:

**Checksum** — download `SHA256SUMS.txt` alongside the artifact and compare:

```bash
# macOS / Linux
shasum -a 256 "Central Tracking Setup 0.4.0.exe"
# Windows (PowerShell)
Get-FileHash "Central Tracking Setup 0.4.0.exe" -Algorithm SHA256
```

The printed hash must match the corresponding line in `SHA256SUMS.txt`.

**Provenance** — verify the binary was built by this repo's release workflow (requires the [GitHub CLI](https://cli.github.com/)):

```bash
gh attestation verify "Central Tracking Setup 0.4.0.exe" --repo michaelklos/central-tracking
```

A corporate environment that blocks unsigned installers can use the verified SHA-256 hash to add a **hash-based AppLocker allow rule**, which doesn't require a code-signing certificate.

## Code Signing

Signing is optional for local use but strongly recommended for distribution: it removes most SmartScreen/Gatekeeper friction and is the single biggest factor in whether locked-down corporate machines (AppLocker publisher rules, Defender ASR prevalence checks) will run the app.

A code-signing certificate is tied to **your identity as the publisher, not to a single project** — one certificate signs as many apps as you want, and SmartScreen reputation accrues to that shared identity.

### macOS

**Prerequisites:** Apple Developer Program membership ($99/yr). There is no free tier for distribution signing.

**Steps:**

1. In Xcode → Settings → Accounts, add your Apple ID and generate a **Developer ID Application** certificate (used for distribution outside the App Store). This adds it to your Keychain automatically.

2. In Keychain Access, find the **Developer ID Application: Your Name (XXXXXXXXXX)** certificate, right-click → Export → save as a `.p12` with a strong password.

3. Base64-encode the `.p12`:
   ```bash
   base64 -i certificate.p12 | pbcopy   # macOS — copies to clipboard
   ```

4. Generate an **app-specific password** for notarization at [appleid.apple.com](https://appleid.apple.com) → Sign-In and Security → App-Specific Passwords.

5. Find your **Team ID** at [developer.apple.com/account](https://developer.apple.com/account) → Membership.

6. Add the following secrets to your GitHub repo (Settings → Secrets and variables → Actions):

   | Secret | Value |
   |---|---|
   | `APPLE_CSC_LINK` | base64-encoded `.p12` from step 3 |
   | `APPLE_CSC_KEY_PASSWORD` | `.p12` export password from step 2 |
   | `APPLE_ID` | your Apple ID email |
   | `APPLE_APP_SPECIFIC_PASSWORD` | app-specific password from step 4 |
   | `APPLE_TEAM_ID` | 10-character Team ID from step 5 |

### Windows

> **Important — this changed in June 2023.** The CA/Browser Forum now requires the private key for all OV and EV code-signing certificates to live on certified hardware (a USB token or a cloud HSM). You can no longer get a plain downloadable `.pfx` for a new certificate, so the old "export a `.pfx`, base64 it into a secret" workflow only applies to legacy file-based certificates. New certificates require either a cloud-signing service or a hardware token, which shapes the options below.

#### How open-source projects usually handle it

For a publicly licensed project (Central Tracking is MIT), the common routes, cheapest first:

| Option | Cost | Notes |
|---|---|---|
| **[SignPath Foundation](https://signpath.org/)** | **Free** for eligible OSS | Donated certificate + managed signing service with a GitHub Actions integration. The standard answer for open-source Windows signing. Requires an OSS license (MIT qualifies), a public repo, and a CI-based build; review/approval takes some lead time. |
| **[Certum Open Source](https://www.certum.eu/)** | ~$30/yr + one-time ~$80–100 for a token/reader (or SimplySign cloud) | A real OV-class certificate issued to you as an individual OSS developer. Covers all your open-source projects, but its terms don't cover closed-source/commercial software. |
| **[Azure Trusted Signing](https://azure.microsoft.com/products/trusted-signing)** | ~$10/mo (~$120/yr) | Microsoft's signing service; key lives in Azure's HSM (no token to ship) with a clean CI story. Identity-based, no open/closed-source restriction, so it also covers any future commercial app. Requires identity verification — orgs typically need 3+ years of operating history; an individual track exists. |
| **Commercial OV** (Sectigo, SSL.com, DigiCert via resellers) | ~$200–350/yr | USB token or cloud-signing add-on (SSL.com eSigner, DigiCert KeyLocker). The "normal business" route. |
| **EV** | ~$300–500/yr | Hardware token + cloud signing for CI. Only worth it for **instant** SmartScreen reputation; OV / Trusted Signing / SignPath have to earn reputation over downloads. |

**Recommendation for this project:** apply to **SignPath Foundation** first (free, purpose-built for OSS). If you'd rather not wait on approval or want something that also covers closed-source work later, **Azure Trusted Signing** (~$120/yr) has the smoothest CI integration; **Certum's open-source cert** (~$30/yr) is the cheapest paid option.

#### Wiring it into CI

The release workflow currently reads `WIN_CSC_LINK` (base64 `.pfx`) and `WIN_CSC_KEY_PASSWORD`. That is the **legacy file-based** path and only works with an old-style exportable certificate. SignPath, Azure Trusted Signing, and token/HSM-backed OV/EV certs each provide their own signing step or GitHub Action that replaces electron-builder's built-in `.pfx` signing — so the `Package (Windows)` job will need updating to match whichever service you choose. Pick a provider first, then adapt the workflow to its documented signing action.
